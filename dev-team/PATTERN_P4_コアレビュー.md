# P4 — コアレビュー（軽量構成）

> **エージェント数**: 2（技術責任者・QA責任者のみ）
> **トークン消費**: ★★☆☆☆
> **推奨場面**: バグ修正・リファクタリング・小規模変更のPRレビュー

---

## 起動するエージェント一覧

| # | 役割 | 使用スキル | 目的 |
|---|------|-----------|------|
| 1 | 技術責任者 | `code-review-tech` | コード品質・設計の妥当性 |
| 2 | QA責任者 | `qa-review` | テストの充足性確認 |

## 除外する役割と理由

| 役割 | 除外理由 |
|------|---------|
| セキュリティ責任者 | 認証・権限・外部入力に触れない変更なら不要 |
| 法務責任者 | 依存関係の変更がなければ不要 |
| 経理責任者 | コスト影響なし |
| プロジェクト責任者 | スケジュール影響なし |
| DevOps責任者 | インフラ変更なし |
| UIデザイナー | UI変更なし |

> **注意**: 以下に該当する場合は P3 以上を使用すること
> - 認証・認可ロジックの変更
> - 外部APIの追加・変更
> - UI/UX の変更
> - インフラ・設定ファイルの変更

---

## 使い方

オーケストレーターへの指示：

```
以下のPRについて、技術責任者とQA責任者の2役割のみで軽量レビューを実施してください。

PR: [PR番号/ブランチ名]
変更内容: [バグ修正 / リファクタリング / テスト追加 等]

起動するエージェント:
1. 技術責任者 → code-review-tech
2. QA責任者   → qa-review

各役割の判定とマージ可否を返してください。
セキュリティ・法務・インフラ観点の問題を発見した場合は P3/P2 へのエスカレーションを推奨してください。
```

---

## 判定ルール
- **2役割とも Approve** → マージ可
- **Request Changes あり** → マージ不可
- **セキュリティ・法務リスクを発見** → P3 または P2 へエスカレーション

---

## エスカレーション基準

```
P4 → P3 へ格上げ:
  □ セキュリティ関連コードに触れている
  □ 新しいUIコンポーネントを追加した
  □ 外部ライブラリを追加した
  □ 想定外の設計変更が含まれていた
```

---

## P3との使い分け
| 変更内容 | 推奨パターン |
|---------|------------|
| 機能追加・UI変更 | P3 開発レビュー |
| バグ修正・リファクタリング | **P4 コアレビュー** ← |
| タイポ修正・ドキュメント更新 | P5 クイックチェック |
