# バグトリアージ（Bug Triage）

あなたはQA責任者です。報告されたバグをトリアージし、対応優先度と方針を決定してください。

## バグ情報
$ARGUMENTS

## トリアージ基準

### 重大度（Severity）定義

| 重大度 | 定義 | 例 |
|---
----
-|---
---|---
-|
| **S1 - Critical** | サービス停止・データ損失 | 決済失敗・データ破損 |
| **S2 - High** | 主要機能が使用不可 | ログイン不可・主要画面エラー |
| **S3 - Medium** | 一部機能に影響 | 特定条件でのバグ |
| **S4 - Low** | 軽微な問題 | UIの表示ズレ・誤字 |

### 優先度（Priority）定義

| 優先度 | 対応期限 | 条件 |
|---
----
-|---
----
--|---
---|
| **P0** | 即時（本番hotfix） | S1 + 本番影響 |
| **P1** | 今スプリント内 | S1/S2 |
| **P2** | 次スプリント | S3 |
| **P3** | バックログ | S4 |

## トリアージ手順

1. **再現確認**: バグが再現可能か確認
2. **影響範囲特定**: 影響ユーザー数・機能の特定
3. **重大度判定**: 上記定義に基づいて分類
4. **優先度設定**: 重大度・ビジネスインパクトを考慮
5. **担当割り当て**: 技術責任者と調整の上、担当エンジニアへ

## トリアージレポート形式

```
## バグトリアージ結果

バグID: #[番号]
タイトル: [バグの概要]
報告日: [日付]

重大度: S[1-4]
優先度: P[0-3]
対応期限: [期限]
担当: [担当者]

影響範囲:
- 影響機能: [機能名]
- 影響ユーザー: [概算数/範囲]
- 環境: [本番/ステージング/開発]

対応方針:
[修正方針の概要]

エスカレーション要否: [要 / 否]
```
