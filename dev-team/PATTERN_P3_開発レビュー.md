# P3 — 開発レビュー（標準構成）

> **エージェント数**: 4（技術・セキュリティ・QA・UI）
> **トークン消費**: ★★★☆☆
> **推奨場面**: 機能開発中のPRレビュー・フィーチャーブランチのマージ前確認

---

## 起動するエージェント一覧

| # | 役割 | 使用スキル | 目的 |
|---|------|-----------|------|
| 1 | 技術責任者 | `code-review-tech` / `architecture-review` | 設計・コード品質 |
| 2 | セキュリティ責任者 | `code-review-security` / `vulnerability-check` | セキュリティリスク |
| 3 | QA責任者 | `qa-review` | テストカバレッジ・品質 |
| 4 | UIデザイナー | `ui-review` / `accessibility-check` | UI変更がある場合のデザイン確認 |

## 除外する役割と理由

| 役割 | 除外理由 |
|------|---------|
| 法務責任者 | 新規ライブラリ追加がなければ不要 |
| 経理責任者 | 開発中のPRにコスト判断は不要 |
| プロジェクト責任者 | スケジュール管理はスプリントイベントで実施 |
| DevOps責任者 | インフラ変更がなければ不要 |

> **条件付き追加**: 新規 npm/pip パッケージ追加 → 法務責任者を追加
> **条件付き追加**: インフラ・CI/CD変更 → DevOps責任者を追加

---

## 使い方

```
/pr-review [PR番号 or ブランチ名]
```

または以下をオーケストレーターへの指示として使用：

```
以下のPRについて、4役割のエージェントを並列起動してレビューしてください。

PR: [PR番号/ブランチ名]

起動するエージェント:
1. 技術責任者       → code-review-tech / architecture-review
2. セキュリティ責任者 → code-review-security / vulnerability-check
3. QA責任者         → qa-review
4. UIデザイナー     → ui-review / accessibility-check（UI変更がある場合のみ）

各役割の判定と、マージ可否の最終判定を返してください。
```

---

## 判定ルール
- **全員 Approve** → マージ可
- **Comment のみ** → 条件付きマージ可（任意対応）
- **1人でも Request Changes** → マージ不可

---

## P2との使い分け
| 状況 | 推奨パターン |
|------|------------|
| 本番リリース直前 | P2 リリース前チェック |
| 機能開発中のPRマージ | **P3 開発レビュー** ← |
| バグ修正・軽微な変更 | P4 コアレビュー |
