# Claude Code プロジェクトルール — skills_hinagata

## バグ・不具合の自動管理ルール

### ルール1: バグ発見時は必ず即時報告する

システム構築・実行中に以下のいずれかを発見した場合、**作業を止めてすぐに `/バグ報告` を実行すること**。
ユーザーへの報告より先に、まず Issue を作成する。

**バグとみなす条件:**
- Bash コマンドが非ゼロの終了コードを返した（意図的な確認コマンドを除く）
- エラーメッセージ・スタックトレースが出力された
- テストが失敗した
- 期待した動作と実際の動作が異なった
- セキュリティ上の問題を発見した
- データ不整合・ファイル破損を発見した

### ルール2: バグ報告の形式

`/バグ報告` を呼び出す際は以下の情報を引数として渡すこと:

```
/バグ報告 [タイトル] | [発生箇所] | [エラー内容] | [重大度: critical/high/medium/low]
```

### ルール3: バグ解決時は必ずステータスを更新する

バグを修正した後は **`/バグ解決`** を実行して Issue をクローズすること。
修正内容・根本原因・再発防止策を必ず記録する。

### ルール4: 修正前に Issue 番号を確認する

既存のバグを修正する場合は、先に `/バグ一覧` で Issue 番号を確認し、
修正中は Issue にコメントを追加しながら作業すること。

### ルール5: 重大度別の対応速度

| 重大度 | 対応 |
|--------|------|
| critical | 即時対応。他の作業をすべて中断して修正する |
| high | 現在のタスク完了後すぐに修正する |
| medium | 今スプリント内に修正する |
| low | バックログに追加して計画的に修正する |

## その他の開発ルール

- DEVELOPMENT_RULES.md のルールも遵守すること
- PR作成前は必ず `/pr-review` を実行すること
- 本番デプロイ前は必ず `/release-gate` を実行すること
